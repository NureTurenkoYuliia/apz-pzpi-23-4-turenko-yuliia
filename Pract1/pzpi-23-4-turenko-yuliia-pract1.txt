3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Загальні відомості про шаблони (патерни) проєктування
     Патерни проєктування – це узагальнені рішення типових задач проєктування програмного забезпечення. Вони описують підхід до організації класів та об’єктів, що дозволяє створювати гнучкі, масштабовані та підтримувані системи.
     У класичній класифікації книги Gang of Four патерни поділяються на три основні групи:
* Породжувальні (Creational) відповідають за створення об’єктів; 
* Структурні (Structural) описують композицію класів та об’єктів;
* Поведінкові (Behavioral) визначають взаємодію між об’єктами.
     Область застосування патернів – проєктування складних систем, де важливі гнучкість, повторне використання коду та можливість легкої модифікації.
     3.2 Опис обраного патерна
     Назва патерна: Декоратор (Decorator).
     Тип патерна: структурний.
     Основне призначення: Decorator вирішує проблему розширення функціональності об’єкта без зміни його коду та без використання складної ієрархії наслідування. Замість створення великої кількості підкласів для кожної комбінації поведінок, патерн дозволяє комбінувати функціональність під час виконання програми і додавати нові можливості поверх існуючого об’єкта.
     Передумови застосування:
* потрібно динамічно додавати або змінювати поведінку об’єкта;
* використання наслідування призводить до великої кількості класів;
* необхідно комбінувати різні варіанти функціональності;
* потрібно розширювати об’єкти без зміни їхнього вихідного коду.
     Переваги:
* гнучке розширення функціональності під час виконання;
* заміна наслідування композицією;
* можливість комбінування різних декораторів;
* зменшення кількості підкласів;
* дотримання принципу Open/Closed.
     Недоліки: 
* ускладнення структури програми (багато дрібних об’єктів);
* важче відслідковувати поведінку через багаторівневе обгортання;
* складність дебагу;
* порядок застосування декораторів може впливати на результат.
     3.3 Структура шаблона (патерна)
     Опис структурних частин:
* Component (Компонент) – частина, що визначає базову операцію;
* Concrete Component (Конкретний компонент) – клас, що реалізує базову функціональність об’єкта;
* Base Decorator (Базовий декоратор) – абстрактний клас, який перенаправляє всі запити вкладеному об'єкту, забезпечуючи однаковий інтерфейс для клієнта; 
* Concrete Decorator (Конкретний декоратор) – клас, який розширює або змінює функціональність і викликає метод обгорнутого об’єкта.
     Взаємодія об’єктів відбувається наступним чином:
* клієнт працює з об’єктами через інтерфейс Component;
* Concrete Component реалізує базову поведінку;
* Decorator обгортає Component;
* Concrete Decorator викликає метод обгорнутого об’єкта й додає нову поведінку до або після виклику.
     Була розроблена UML-діаграма класів на прикладі повідомлень (рис. 1.1), яка відображає структуру патерна Decorator. Вона показує спільний інтерфейс компонентів, базові реалізації та ієрархію декораторів. Ключовим є зв’язок композиції, де декоратор містить інший об’єкт того ж типу, що дозволяє динамічно розширювати поведінку.

Рисунок 1.1 – UML-діаграма класів
     
     В діаграмі серед компонентів є:
* INotificationService – спільний інтерфейс для всіх компонентів;
* Сервіси, що реалізують інтерфейс (EmailNotificationService, SmsNotificationService, PushNotificationService);
* NotificationDecorator – декоратор як абстракція;
* Конкретні декоратори, які наслідують базовий декоратор (LoggingNotificationDecorator, RetryNotificationDecorator);
* Клієнт (виклик функцій), що працює через інтерфейс, а не конкретні класи.
     Також для кращого розуміння того, як виклики проходять через ланцюг декораторів, була розроблена діаграма послідовності (рис. 1.2). По ній можна побачити, що кожен об’єкт додає власну поведінку до або після виклику базового компонента, що і є основною ідеєю патерна Decorator.

Рисунок 1.2 – UML-діаграма послідовності
     
     В діаграмі Client викликає Send() у першого декоратора (LoggingDecorator), який виконує дію до (логування) і передає виклик далі. RetryDecorator починає спробу та викликає сервіс, тобто EmailService. Він виконує реальну відправку повідомлення і повертає результат. А назад по ланцюгу RetryDecorator завершує роботу і LoggingDecorator виконує дію після.
     3.4 Реалізація шаблона (патерна)
     Реалізація патерна виконана мовою C#, як приклад для системи, де потрібно надсилати відмінні повідомлення користувачам (код В.1 – В.8). Система повинна підтримувати різні канали (Email, SMS, Push), легко розширюватися новими каналами і додавати поведінку (логування, повторні спроби) без зміни основного коду.
     Основні етапи реалізації патерну Decorator:
* створюється спільного інтерфейсу або абстрактного класу, який описує базову операцію для всіх об’єктів;
* реалізуються класи з основною функціональністю без додаткових розширень;
* створюється абстрактний клас (декоратор), який реалізує той самий інтерфейс і містить посилання на об’єкт цього інтерфейсу;
* створюються конкретні декоратори, які наслідують базовий декоратор і додають нову поведінку до або після виклику основного методу;
* композиція об’єктів – базові об’єкти обгортаються одним або кількома декораторами, формуючи ланцюг викликів;
* використання через інтерфейс: клієнт працює з об’єктами через спільний інтерфейс, не знаючи, чи це базовий об’єкт, чи декорований.

В.7 Клас RetryNotificationDecorator
1  public class RetryNotificationDecorator : NotificationDecorator
2  {
3      private readonly int _maxRetries;
4  
5      public RetryNotificationDecorator(INotificationService service, int maxRetries = 3) 
6          : base(service)
7      {
8          _maxRetries = maxRetries;
9      }
10 
11     public override void Send(string message)
12     {
13         int attempt = 0;
14 
15         while (true)
16         {
17             try
18             {
19                 attempt++;
20                 Console.WriteLine($"[Retry] Attempt {attempt}");
21 
22                 base.Send(message);
23 
24                 break;
25             }
26             catch (Exception)
27             {
28                 if (attempt >= _maxRetries)
29                 {
30                     Console.WriteLine("[Retry] Failed after max attempts");
31                     throw;
32                 }
33 
34                 Console.WriteLine("[Retry] Retrying...");
35                 Thread.Sleep(500);
36             }
37         }
38     }
39 }
